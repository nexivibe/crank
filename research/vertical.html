THE EASY WIN

SCP & SFTP
RECOMMENDED

SFTP

One dependency unlocks the full SFTP subsystem. Channels multiplex over the existing ClientSession -- no new TCP connection, no new SSH handshake, no new authentication. 

<dependency>
  <groupId>org.apache.sshd</groupId>
  <artifactId>sshd-sftp</artifactId>
  <version>${sshd.version}</version>
</dependency>
  • ▶ Full directory listing, traversal, creation, removal
  • ▶ Random-access read/write (seek to offset)
  • ▶ File stat (size, mtime, permissions)
  • ▶ SFTP protocol versions 3-6 with auto-negotiation
  • ▶ Java NIO FileSystem integration
  • ▶ Rename, symlink, readlink
SECONDARY

SCP

Simpler protocol for basic copy operations. Deprecated by OpenSSH 9.0+ (now uses SFTP internally). Consider only for the remote-to-remote transfer helper. 

<dependency>
  <groupId>org.apache.sshd</groupId>
  <artifactId>sshd-scp</artifactId>
  <version>${sshd.version}</version>
</dependency>
  • ▶ ScpTransferEventListener for progress
  • ▶ Remote-to-remote copy helper
  • ▶ No directory listing
  • ▶ No random access / resume
  • ▶ No NIO FileSystem integration
  • ▶ Deprecated upstream

03

VS RSYNC

COMPETITIVE ANALYSIS

HOW RSYNC WORKS

rsync is the gold standard. Its killer feature is the rolling checksum delta transfer algorithm: for a 1GB file with a 1KB change, rsync transfers ~one block (not 1GB).

STEP 1

Receiver splits its copy into fixed blocks, computes weak + strong checksums per block

STEP 2

Sender scans its file with a sliding window matching against those checksums

STEP 3

Only non-matching literal data + block references are transmitted

STEP 4

Receiver reconstructs the file from its blocks + literal data

RSYNC ADVANTAGES

  • + Block-level delta transfer
  • + --delete for removing stale files
  • + --partial for resuming transfers
  • + --checksum integrity verification
  • + Battle-tested, universally available

RSYNC DISADVANTAGES

  • - Requires rsync binary on remote server
  • - Not a library -- standalone process
  • - No Java API (rsync4j is JNI, not pure JVM)
  • - Complex protocol, tightly coupled transport
  • - New SSH connection per invocation

05

DELTA TRANSFER

THE HARD PROBLEM

The critical question: Is block-level delta transfer achievable without requiring anything on the remote server?

A

SFTP-ONLY DELTA

NO REMOTE TOOLING
01

Read remote file in blocks via SFTP seek + read, compute rolling + strong checksums locally for each block. Cache signatures.

02

Run rolling checksum sliding window over local file, matching against remote block signatures.

03

Use SFTP seek + write to overwrite only the changed blocks on the remote file.

Tradeoff: First sync downloads all remote data to compute signatures. Subsequent syncs use cached signatures.
B

SSH EXEC CHECKSUM HELPER

LIGHTWEIGHT REMOTE

Use SSH exec channels to run checksum commands on the remote -- no file download needed:

// Fast: get file checksum without downloading
val channel = session.createExecChannel("sha256sum /remote/file.txt")

// Or use a Python one-liner for block signatures
val channel = session.createExecChannel("""
  python3 -c 'import hashlib,sys
  bs=65536
  with open(sys.argv[1],"rb") as f:
    while b:=f.read(bs):
      print(f"{sum(b)&0xFFFFFFFF},{hashlib.md5(b).hexdigest()}")
  ' /remote/file.txt
""")
Available on >99% of Linux servers. No deployment, no installation.
C

HYBRID

RECOMMENDED
FIRST

Try remote checksumming via SSH exec -- fast, bandwidth-efficient

FALLBACK

SFTP-only delta if remote tools unavailable -- universal compatibility

SKIP

Transfer small files (<1MB) in full via SFTP -- not worth delta overhead

KEY DESIGN DECISIONS

01 SFTP over SCP

SCP is deprecated by OpenSSH. SFTP provides directory ops, random access, and NIO FileSystem integration.

02 Connection reuse is non-negotiable

Sync rides the existing terminal session. No new connections, no new auth, no new host key verification.

03 Delta transfer is optional but strategic

File-level sync covers most use cases. Delta transfer is the differentiator that makes Crank competitive with rsync.

04 Build rolling checksum in Kotlin

Jarsync is archived. Yajsync is GPL. The algorithm is ~500 lines and well-documented. Own it.

05 Hybrid remote checksumming

Try SSH exec first (fast, no download), fall back to SFTP block reads (universal). Degrade gracefully.

06 Bidirectional sync is the moat

rsync is unidirectional. Unison has no GUI. Crank owns the bidirectional + visual conflict resolution niche.

09

CONCLUSION

Adding file synchronization to Crank is a high-leverage vertical integration. The hardest part (persistent SSH session management) is already done. The SFTP transport is one Maven dependency away. The investment to reach rsync-competitive capability is 8-12 weeks, with useful milestones at every phase.

Crank becomes the only tool that combines multi-session SSH terminal management with integrated, bidirectional, delta-efficient file synchronization -- all in a single pure-JVM application with a visual interface.

RECOMMENDED STARTING POINT

Add sshd-sftp to pom.xml, create a sync/ package under ape.crank, and build Phase 1 (file browser + basic transfer). Ship it. Then iterate toward delta sync based on user demand.
